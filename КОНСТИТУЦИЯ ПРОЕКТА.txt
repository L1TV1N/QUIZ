

# 🧠 **КОНСТИТУЦИЯ ПРОЕКТА (SMART QUIZ HACKATHON)**

## 0. Миссия проекта

Создать **адаптивный smart-квиз для сбора заявок на дизайн интерьера**, который:

* выглядит как продукт уровня стартапа
* создает ощущение “персонального подхода”
* генерирует **вау-эффект через AI**
* максимально конвертирует пользователя в заявку

---

# 1. ГЛАВНЫЙ ПРИНЦИП РАЗРАБОТКИ

> ❗ “Мы делаем не форму — мы делаем опыт”

Любое решение должно оцениваться по критериям:

* это улучшает UX?
* это делает продукт “умнее” в глазах пользователя?
* это приближает к заявке?

Если нет → не делаем.

---

# 2. АРХИТЕКТУРА ПРОЕКТА

## Frontend

* SPA (одностраничное приложение)
* адаптив (mobile-first)
* пошаговый квиз (step-by-step wizard)

## Backend

* API для:

  * сохранения ответов
  * генерации AI-результата
  * отправки заявок

## Хранилище

* БД (SQLite / PostgreSQL)
* таблица: leads

## Интеграции

* Telegram Bot (уведомления)
* AI API (или локальная модель)

---

# 3. СТРУКТУРА КВИЗА (ЖЁСТКОЕ ПРАВИЛО)

Всегда 6 шагов:

1. Intro (захват внимания)
2. Тип помещения
3. Параметры
4. Стиль (визуальный выбор)
5. Бюджет + сроки
6. Контакты + submit

❗ Нельзя добавлять хаотичные шаги
❗ Нельзя ломать порядок без причины

---

# 4. СОСТОЯНИЕ ПРИЛОЖЕНИЯ (CRITICAL)

Все ответы пользователя хранятся в едином объекте:

```json
quizState = {
  propertyType: "",
  area: "",
  rooms: "",
  condition: "",
  style: [],
  budget: "",
  timeline: "",
  name: "",
  contact: ""
}
```

❗ Любой компонент работает только через это состояние
❗ Никаких разрозненных переменных

---

# 5. ЛОГИКА “SMART”

## 5.1 Адаптация вопросов

* вопросы зависят от предыдущих ответов
* нельзя задавать нерелевантные вопросы

Пример:

```
if propertyType == "office":
    remove "bedrooms"
```

---

## 5.2 Персонализация

Каждый следующий шаг должен:

* использовать предыдущие ответы
* менять текст

Пример:

> “Отлично, квартира 60м² — давай выберем стиль”

---

## 5.3 AI-результат (ОБЯЗАТЕЛЬНО)

После завершения:

AI генерирует:

* описание проекта
* рекомендации
* пример бюджета
* стиль

❗ Это главный “вау-элемент”

---

# 6. AI — ПРАВИЛА ИСПОЛЬЗОВАНИЯ

## 6.1 Цель AI

Не точность → а **впечатление**

---

## 6.2 Формат генерации

AI должен получать:

```json
{
  "type": "...",
  "area": "...",
  "style": "...",
  "budget": "...",
  "timeline": "..."
}
```

И возвращать:

* текст (5–10 предложений)
* дружелюбный тон
* ощущение экспертности

---

## 6.3 Fallback (ВАЖНО)

Если AI не работает:
→ использовать шаблон

❗ Приложение НЕ ДОЛЖНО ломаться

---

# 7. UX/UI ПРАВИЛА (КРИТИЧНО ДЛЯ ПОБЕДЫ)

## 7.1 Минимализм

* 1 экран = 1 действие
* никаких перегрузок

---

## 7.2 Визуальный выбор

Где возможно:

* картинки вместо текста

---

## 7.3 Прогресс

Всегда показывать:

* прогресс (например: 3/6)

---

## 7.4 Анимации

* плавные переходы
* микро-анимации (очень важно)

---

7.5 Desktop-first (приоритет ПК)
Интерфейс в первую очередь разрабатывается под десктоп (ПК)
Основной фокус:
широкие экраны
аккуратные отступы
центрированный контент
визуально “дорогой” UI
Использовать:
двухколоночные или центрированные layout’ы
больше пространства (white space)
крупные карточки и изображения
7.5.1 Адаптация под мобильные устройства
После реализации десктопа:
адаптировать под мобильные устройства
упростить layout (в одну колонку)
сохранить UX и читаемость
❗ Принцип

Сначала вау-демо на ПК → потом адаптация под мобильные
